## 🎯 プロジェクト概要

本プロジェクトは、Unreal Engine 5の最新アーキテクチャ（Lyra Starter Game）とGameplay Ability System（GAS）の設計思想をベースに構築した、拡張性が高く疎結合なモジュラー型ゲームフレームワークです。

ゲーム開発において「強固な土台を作り、適切なドメインで責務を分割し、複数プロジェクトで使い回す」ことは、非常に重要なアプローチだと考えています。
本フレームワーク（Game Core Framework: GCF）は、UE5のモダンな機能（Modular Gameplay, GameFeatures等）を前提とした「4層レイヤーアーキテクチャ」における第2層（中間基盤）として位置づけています。

### 🏗️ 4層アーキテクチャという考え方

1. **Engine Layer（Unreal Engine）**  
   物理演算、描画、メモリ管理などを担うインフラ層。固有のゲームルールを持たない、あらゆるプロジェクトで共有される基盤。
2. **Core Framework Layer（本フレームワーク：GCF）**  
   エンジンの上に構築される中間層。入力のルーターやキャラクターのインターフェースなど、**抽象的な骨組み（Skeleton）のみ**を提供し、ゲーム固有の仕様（シューター、RPG等）は持たないことを意図しています。複数プロジェクト間で使い回される「第二の基盤」として設計しました。
3. **Modular Feature Layer（GameFeatures等）**  
   第2層のルールに従って実装された、「付け替え可能な具体のパッケージ」。独立したドメインごとにパッケージ化されたモジュール（基本移動、銃火器、魔法など）を、要件に合わせて動的に注入（Inject）する層です。
4. **Project / Experience Layer（最終的なゲームの具体）**  
   用意された第3層のモジュール（カセット）を組み合わせ、「このマップでは『基本移動』と『銃』を有効にする」といった最終決定を下し、ゲーム体験（Experience）を組み立てる層です。

### 🧩 なぜ「Lyra」の構成ではなく「GCF」が必要と感じたか

Epic Gamesが提供する『Lyra Starter Game』はアーキテクチャのお手本として非常に優秀だと感じています。
一方で、汎用的な中間層として別ジャンルのゲームにそのまま転用しようとすると、「**シューターという具体に寄りすぎている（レイヤー3・4の要素がコアに混ざっている）**」という課題があると感じました。

TPS専用の仕様や武器の概念が密結合したまま別ジャンルのゲームを作ろうとすると、不要な要素を剥がす作業でアーキテクチャが崩れていく経験をしてきました。
そこで、「**Lyraの設計思想（モジュール化とデータ駆動）の真髄だけを参考にし、ゲーム固有の具体を排除した抽象基盤（GCF）**」をゼロから構築することにしました。

### 🌟 このフレームワークを通して目指している姿

目先の開発に追われるだけでなく、長期的な視点で「開発効率と質の高いゲーム体験」を両立させることが理想だと考えています。

* **プロジェクト固有の「具体」への集中:**  
  汎用的な中間層が整備されることで、チームはインフラ構築の負担から徐々に解放され、そのゲーム特有の「面白さ」の実装により集中しやすくなると考えています。
* **適切なドメイン分離による再利用の促進:**  
  機能を「どのドメイン（責務）に属するか」を意識して切り出すことで、システム全体の疎結合化と知見の再利用につながると考えています。
* **組み換えによるイテレーションの加速:**  
  ドメインが分離されカセット化されていることで、プランナーやレベルデザイナー等の非プログラマでも機能を組み替えてテストしやすくなり、試行錯誤のサイクルが速まることを期待しています。

## 💡 コアとなる設計思想

- **明確な関心の分離**  
ACharacter や APlayerController へのハードコーディングを排除し、各機能を独立したコンポーネントとして分割。既存のコードを汚染することなく、新機能やキャラクターを安全に追加・拡張できます。

- **魂と肉体の分離**  
PlayerState（永続的な魂）と Pawn（一時的な肉体）の責務を明確に分離。これによりAbilityやInputのバインディングを「永続的なもの」と「一時的なもの」に分解して定義することで、ポゼッション状態に左右されない柔軟なシステムを実現します。

- **データドリブンなルーティング**  
プログラマ以外のメンバーでも、DataAssetを編集するだけで入力とアビリティの紐付けや動作調整ができるルーティング層を構築。チーム全体のイテレーション速度を向上させます。

- **安全な非同期ライフサイクル管理**  
` GameFrameworkComponentManager` （GFCM）を活用し、各コンポーネントの依存関係と初期化状態（Feature State）を厳密に管理。イベント駆動と状態管理のハイブリッド構成を実装することにより、常に最新の状態を同期することができ、「参照先のものがまだ生成されていない」というUE特有の初期化順序問題をシステムレベルで防いでいます。

- **パフォーマンス指向のTickレス設計**  
毎フレームのTick処理への依存を原則として排除し、イベント駆動（Event-Driven）と状態のブロードキャストによってシステムを稼働させています。GameスレッドのCPUオーバーヘッドを抑制し、高いランタイムパフォーマンスを維持します。

> 💡 **参考記事（アーキテクチャの背景）**  
> コードを読む前に、本フレームワークの根底にある「**予測不能なゲームプレイを生むための疎結合**」という設計哲学について、以下の技術記事で解説しています。開発の過程で直面した強烈なトレードオフと、その解決策に興味がある方はぜひご一読ください。  
> 🔗 [なぜ私はゲーム開発で「疎結合」と「コンポーネント指向」に異常なほどこだわるのか](https://zenn.dev/munimaru62o/articles/c6ed730c6e4c61)

GCFの設計思想や各システムの詳細については、以下のドキュメントを参照してください。

- **[アーキテクチャ概要](Document/ja/Architecture/overview.md)**  
  マルチプレイ開発で頻発する「初期化のレースコンディション」や「責務の肥大化」をシステムレベルで防ぐための根幹思想（魂と肉体の分離など）と、実務の失敗から得た教訓をまとめています。

- **[GAS統合とAbilityルーティング (Dual ASC & Router パターン)](Document/ja/Architecture/ability_system.md)**  
  PlayerStateとPawnの両方にASCを持たせる高度な「Dual ASC構成」を採用。入力とアビリティの密結合を排除し、タグのプレフィックスで発動先を動的に切り替えるルーティング機構を解説しています。

- **[インプットシステム (InputBridge & Manager パターン)](Document/ja/Architecture/input_system.md)**  
  ポゼッション時の非同期ロードによって発生する入力バインドのクラッシュを防ぐため、状態が整うまで要求を待機し、安全なタイミングで一斉適用する堅牢なマネージャー設計です。

- **[アクター制御システム (Interface駆動の意図伝達 & オプトイン設計)](Document/ja/Architecture/control_system.md)**  
  「プレイヤーの操作意図」と「Pawnの物理挙動」をインターフェースで明確に分離。入力側が意図をPushし、Pawnがキャッシュし、Mover等の物理エンジン側がPullして翻訳する「意図のバケツリレー」による高度な疎結合アーキテクチャを解説しています。

## 🚀 ソースコードへのアクセスとインストール

### Step 1: Epic GamesとGitHubの連携（アクセス権の取得）

ソースコードおよびプロジェクト本体は、**Epic公式ネットワーク上のPrivate Fork**（[https://github.com/munimaru62o/UnrealEngine.git](https://github.com/munimaru62o/UnrealEngine)）にホスティングされています。

アクセス権は招待制ではありません。Epic Gamesが提供する **[Unreal Engineのソースコードをダウンロードする手順](https://www.unrealengine.com/ue-on-github)** に従ってアカウント連携を完了し、Epic GamesのGitHub Organization（`@EpicGames/developers`）に参加した状態となれば、URLから直接アクセスできるようになります。

### Step 2: リポジトリの取得（Clone and Play）

連携が完了したら、以下のURLから専用のブランチ（`gcf-release`）へアクセスし、クローンすることが可能になります。
（※本フレームワークは拡張性を最大化するためLyraの基盤プラグインに依存していますが、面倒な手動でのコピー手順を省くため、**必須プラグイン群はすべてリポジトリ内に同梱**されています。クローンしてすぐに遊べる「Clone and Play」構成となっています！）

```bash
# gcf-release ブランチのみを軽量にクローンする
git clone -b gcf-release --single-branch https://github.com/munimaru62o/UnrealEngine.git
```

### Step 3: プロジェクトのビルド

1. クローンしたリポジトリのルートにある `GCF_SampleProject.uproject` を右クリックし、「Generate Visual Studio project files」を実行します。
2. 生成された `.sln`（またはIDEのプロジェクトファイル）を開き、プロジェクトをビルド（Development Editor等）してください。
3. エディタが起動したら、`Edit > Plugins` から本プラグインおよび依存プラグイン（GameplayAbilities, EnhancedInput, Mover など）が有効化（Enabled）されていることを確認してください。

### ご自身のプロジェクトへ導入（移植）する場合

このフレームワークをご自身のゲームプロジェクトに導入したい場合は、クローンしたリポジトリ内の `Plugins` フォルダにある **`GameCoreFramework` 本体** と、同梱されている **Lyra依存プラグイン群（`CommonGame`等）** をまとめて、ご自身のプロジェクトの `Plugins` フォルダへそのままコピーしてください。

## 🎬 デモ & サンプル

本プロジェクトには挙動を確認するための、デモ用アセットを用意してあります。  
C++のコードを変更せずとも、DataAssetを編集するだけで動的にアビリティや挙動を切り替えられる設計になっています。

```text
GameCoreFramework/Content/Sample
├── Assets/         # PawnやParticleのためのアセット類
├── Blueprints/     # PawnやAbilityのBPクラス群
├── DataAssets/     # データ駆動のコア部分、ここに各DataAssetが定義されています
├── Experiences/    # Experience定義
└── Maps/           # デモ用マップ定義
└── UI/             # デバッグHUD用UI定義
```

> **⚠️ 補足: Experienceのロード機能について**   
> 本フレームワークは、コア基盤の堅牢性を重視した最小構成（MVP）として構築されているため、現時点ではUIを介した動的なExperienceの切り替え機能は実装されていません。Experienceの指定・変更は、対象マップの `World Settings` 内にある「Default Gameplay Experience」からのみ行う設計となっています。

### 🖥️ Sample動画  

https://github.com/user-attachments/assets/4acb12b9-a0ea-425b-9c0f-9894a5bd1f55

#### **💡 本動画の見どころ**
ポゼッション（憑依）を変更した瞬間に、古い肉体の入力バインドが安全に破棄され、**新しいPawnの `InputBinding` が動的に更新される様子**や、新しい肉体に付与されたAbilityが即座に有効化・ルーティングされる「ライフサイクルの安全な同期」を確認できます。

#### 📊 画面上のデバッグ表示について
- **Debug Input Info:** 現在アクティブにバインドされている入力アクションとルーティング状態をリアルタイムに表示します。
- **Debug State Info:** GFCMの `InitState`（各機能の初期化フェーズ）や、現在のポゼッション状態を監視します。
- **Debug Log:** ポゼッション切り替えイベントや、Abilityルーター経由のタグ送信ログを出力します。

#### 🎮 ポゼッションの実行とターゲット選定
- 画面内でハイライト（アウトライン表示）されたActorに対して、`Interact` アビリティ（魂側の永続能力）を実行することでポゼッションを移行します。
- ターゲットはカメラモードに応じて動的に選定されます。（TPSモード時は画面中央、見下ろしモード時はマウスカーソルの位置を基準に判定）。

#### 🤖 操作可能なActorのバリエーション（新旧システム・物理エンジンの共存）
アーキテクチャの疎結合性を証明するため、全く異なる物理コンポーネントを持つPawn間をシームレスに行き来します。
- **白いマネキン:** UE5の次世代システム `Mover` プラグインをベースにした本フレームワーク独自の `GCFCharacterMover` による移動と、Jump / Crouch 機能。
- **色付きマネキン:** 従来の `CharacterMovementComponent` を使用した移動制御に加え、現在の「肉体」に固有で定義された専用アビリティの実行機能。
- **球体 (Mover):** UE5の次世代システム `Mover` プラグインによるTickベースの移動制御と、見下ろし視点カメラ。
- **車両 (Chaos Vehicle):** `ChaosVehicleMovementComponent` による本格的な車両制御と、Headlight / Handbrake などの固有操作。

### 🌐 Client Sample動画（Network Lag Simulation）  

https://github.com/user-attachments/assets/7a6cf746-329a-4140-8461-0b7f703d57c6

#### **💡 本動画の見どころ**
劣悪な通信環境下において、通信遅延による「初期化順序の逆転」や「非同期ロード時のライフサイクルのズレ（レースコンディション）」が発生しても、GFCMによる厳格な状態管理がそれらを安全に吸収している様子が確認できます。

#### 📡 テスト環境（意図的なネットワーク遅延）
Listen Serverに接続したClient環境にて、マルチプレイで頻発する過酷なネットワーク制限をエミュレートしています。
- **Latency (Ping遅延):** 200 ms
- **Packet Loss (パケットロス):** 10%

このような厳しいラグ環境下においても、ポゼッションの移行が安全に完了し、Null参照によるクラッシュや入力のバインド漏れの発生を設計レベルで抑え込み、アーキテクチャが堅牢に機能していることが証明されています。
